# teton-challenge

TPM-backed Wi-Fi provisioning for Linux IoT devices. A device running in SoftAP mode serves a TLS-protected credential form. The TLS certificate is signed by a per-device CA and the private key never leaves the TPM.

---

## Roles

This demo involves two machines:

| Role | Description |
|---|---|
| **Device** | The Linux machine being provisioned. Runs the provisioning software, broadcasts the SoftAP, and serves the credential form. Must be Ubuntu 22.04 LTS or Raspberry Pi OS Bookworm with a Wi-Fi interface. |
| **Configurator** | The machine used to submit Wi-Fi credentials — any laptop, phone, or tablet with a browser. Must connect to the Device's Wi-Fi AP during provisioning. |

For a single-machine demo (device and configurator on the same Ubuntu machine), all steps run on that one machine and `install-ca.sh` handles the browser trust automatically.

---

## Prerequisites

### Device

Raspberry Pi OS Bookworm or Ubuntu 22.04 LTS (x86-64), with a Wi-Fi interface capable of AP mode.

**Required packages** — run on the **Device**:

```bash
sudo apt-get install -y \
  swtpm tpm2-tools tpm2-abrmd \
  libtss2-dev tpm2-openssl \
  openssl \
  python3 python3-pip python3-venv \
  hostapd dnsmasq network-manager \
  libnss3-tools
```

**Python dependencies** — run on the **Device**:

On **Ubuntu 22.04** (Python 3.10), pip installs system-wide directly:

```bash
pip install -r requirements.txt
```

On **Ubuntu 24.04** (Python 3.12+), the OS blocks system-wide pip installs (PEP 668). Use a virtual environment instead:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Re-activate the venv (`source .venv/bin/activate`) in any new terminal before running tests or `provision.py`.

### Configurator

Any device with a browser (Chrome, Firefox, Safari, Edge). No software installation required beyond trusting the CA certificate (see [Install CA certificate](#install-ca-certificate) below).

## Step 1 — One-time setup

> **Run on: Device**

Initialises the software TPM, generates the device key inside it, and issues the device certificate signed by a local Wajntraub demo CA:

```bash
sudo ./scripts/setup.sh
```

What it does:
- Wipes and restarts swtpm at `/tmp/tpm-state`
- Creates a primary key and a persistent RSA-2048 device key at handle `0x81000001`
- Generates `certs/wajntraub-demo-ca.crt` (demo CA) and `certs/device.crt` (device certificate)
- Verifies the cert chain with `openssl verify`

**Cleaning up** — to remove all artefacts created by setup (TPM state, sockets, generated keys and certificates):

```bash
sudo ./scripts/setup.sh --clean
```

The script lists every file it will delete and asks for confirmation before proceeding.

## Step 2 — Install CA certificate

> **Run on: Configurator** (the machine whose browser will open the provisioning page)
>
> If the Device and Configurator are the same machine, run this on that machine.

The configurator browser must trust the Wajntraub demo CA to show a padlock instead of a TLS warning.

**Ubuntu / Debian** — automated:

```bash
sudo ./scripts/install-ca.sh
```

This copies `certs/wajntraub-demo-ca.crt` into `/usr/local/share/ca-certificates/` and runs `update-ca-certificates` (picked up by Chrome and Chromium). It also imports the CA into every Firefox NSS profile found under `/home`.

> `certs/wajntraub-demo-ca.crt` is generated by `setup.sh` on the Device. If the Device and Configurator are different machines, copy this file to the Configurator first.

**Other operating systems** — import `certs/wajntraub-demo-ca.crt` manually:

| OS | Steps |
|---|---|
| Windows | `certmgr.msc` → Trusted Root Certification Authorities → Import |
| macOS | Keychain Access → System → File → Import Items → set Trust to "Always Trust" |
| iOS | AirDrop or email the `.crt` file → Settings → General → VPN & Device Management → Install Profile → trust it |
| Android | Settings → Security → Install from storage → CA certificate |

Restart your browser after importing.

## Step 3 — Run

> **Run on: Device**

**Find your Wi-Fi interface name** — Linux does not always name the Wi-Fi interface `wlan0`. Run:

```bash
ip link
```

Look for an interface that is not `lo` (loopback) or `eth`/`enp` (Ethernet). Common Wi-Fi names are `wlan0`, `wlo1`, `wlp2s0`. Note yours down — you will need it below.

**Run the provisioning daemon:**

If you installed Python dependencies system-wide (Ubuntu 22.04):

```bash
sudo PROVISION_IFACE=<your-interface> python3 device/provision.py
```

If you used a virtual environment (Ubuntu 24.04):

```bash
sudo PROVISION_IFACE=<your-interface> $(pwd)/.venv/bin/python3 device/provision.py
```

> `sudo` strips environment variables, so `PROVISION_IFACE` must be passed inline as shown above, not exported beforehand. An absolute path to the venv Python is required because `sudo` also resets `PATH`. Use `$(pwd)` to avoid typing the full path manually.

Expected terminal output (happy path):

```
2026-03-31 12:00:00 INFO State: INIT
2026-03-31 12:00:01 INFO State: INIT → AP_MODE
2026-03-31 12:00:01 INFO State: AP_MODE → PROVISIONED
2026-03-31 12:00:01 INFO State: PROVISIONED → CONNECTING
2026-03-31 12:00:05 INFO State: CONNECTING → ONLINE
2026-03-31 12:00:05 INFO Provisioning complete — device is connected.
```

**Provisioning steps:**

1. The Device broadcasts a Wi-Fi AP named `Wajntraub-Demo-0000` on the interface you specified
2. On the **Configurator**, connect to the `Wajntraub-Demo-0000` Wi-Fi network
3. On the **Configurator**, open a browser and navigate to `https://setup.wajntraub-demo.local` — a padlock should appear with no TLS warning
4. Enter the target Wi-Fi SSID and password, click **Provision**
5. The Device disconnects from the SoftAP, connects to the target network, and logs `ONLINE`

**Environment variables** (set on the Device before running):

| Variable | Default | Description |
|---|---|---|
| `PROVISION_IFACE` | `wlan0` | Wi-Fi interface to use for SoftAP — override if your interface is named differently |
| `PROVISION_TIMEOUT` | `600` | Seconds to wait for credential submission before giving up |

> **Run on: Device** (or any Ubuntu 22.04 machine with the dev dependencies installed)

**Unit tests only** (no hardware required, always works):

```bash
pip install -r requirements-test.txt
python3 -m pytest tests/unit/ -v
```

**Full suite including integration tests** — requires `tpm2-openssl` and OpenSSL 3. On Ubuntu 22.04 / Raspberry Pi OS Bookworm these are available natively:

```bash
python3 -m pytest tests/ -v
```

**Full suite via Docker** — if `tpm2-openssl` is not installed on your machine (e.g. Debian 11, macOS), use the Docker runner instead. Requires Docker:

```bash
./scripts/docker-test.sh
```

This builds an Ubuntu 22.04 image on first run (cached afterwards) and executes the full suite inside it:

```

```

## Security notes and limitations

**TPM key vs TLS key:** `setup.sh` generates an RSA key inside the TPM (persistent at handle `0x81000001`) to demonstrate the manufacturing provisioning step. However, Python's `ssl.SSLContext.load_cert_chain()` calls OpenSSL's `SSL_CTX_use_PrivateKey_file()`, which uses `fopen()` and cannot resolve TPM store URIs. TLS termination therefore uses a separate software RSA key (`certs/device.key`) generated by `setup.sh`. The TPM key is present but not used for TLS in this demo.

**Production path:** The correct production stack for TPM-backed TLS in Python is `tpm2-pkcs11` (PKCS#11 interface over TPM2) combined with OpenSSL's PKCS#11 engine/provider. This makes the TPM key reachable from Python's ssl module with a `pkcs11:` URI as the keyfile, keeping the private key hardware-backed. This was considered out of scope for this challenge.

**certs/ directory:** All files under `certs/` are generated locally by `setup.sh` and are excluded from version control (`.gitignore`). Never commit private keys or CA credentials.
