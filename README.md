# teton-challenge

Wi-Fi provisioning for Linux IoT devices. A device running in SoftAP mode serves a TLS-protected credential form. The TLS certificate is issued by a per-device CA and verified by the configurator's browser.

---

## Roles

This demo involves two machines:

| Role | Description |
|---|---|
| **Device** | The Linux machine being provisioned. Runs the provisioning software, broadcasts the SoftAP, and serves the credential form. Must be Ubuntu 22.04 LTS or Raspberry Pi OS Bookworm with a Wi-Fi interface. |
| **Configurator** | The machine used to submit Wi-Fi credentials — any laptop, phone, or tablet with a browser. Must connect to the Device's Wi-Fi AP during provisioning. |

For a single-machine demo (device and configurator on the same Ubuntu machine), all steps run on that one machine and `install-ca.sh` handles the browser trust automatically.

---

## Prerequisites

### Device

Raspberry Pi OS Bookworm or Ubuntu 22.04 LTS (x86-64), with a Wi-Fi interface capable of AP mode.

**Required packages** — run on the **Device**:

```bash
sudo apt-get install -y \
  openssl \
  python3 python3-pip python3-venv \
  hostapd dnsmasq network-manager \
  iw \
  libnss3-tools
```

Python dependencies are installed automatically by `setup.sh` into a `.venv` at the project root — no manual `pip install` needed.

### Configurator

Any device with a browser (Chrome, Firefox, Safari, Edge). No software installation required beyond trusting the CA certificate (see [Install CA certificate](#install-ca-certificate) below).

## Step 1 — One-time setup

> **Run on: Device**

Generates the demo CA and issues the device certificate:

```bash
sudo ./scripts/setup.sh
```

What it does:
- Generates `certs/wajntraub-demo-ca.crt` (demo CA) and `certs/device.crt` / `certs/device.key` (device certificate and key)
- Verifies the cert chain with `openssl verify`
- Creates `.venv/` and installs all Python dependencies (Flask, pytest, requests)

**Multiple devices** — each device needs its own `device.key` and `device.crt`, but they can all share the same CA. To issue a new device cert without regenerating the CA (no browser re-import needed):

```bash
sudo ./scripts/setup.sh --new-device-cert
```

**Cleaning up** — to remove all artefacts created by setup (generated keys, certificates, and the virtual environment):

```bash
sudo ./scripts/setup.sh --clean
```

The script lists every file it will delete and asks for confirmation before proceeding.

## Step 2 — Install CA certificate

> **Run on: Configurator** (the machine whose browser will open the provisioning page)
>
> If the Device and Configurator are the same machine, run this on that machine.

The configurator browser must trust the Wajntraub demo CA to show a padlock instead of a TLS warning.

**Ubuntu / Debian** — automated:

```bash
sudo ./scripts/install-ca.sh
```

This copies `certs/wajntraub-demo-ca.crt` into `/usr/local/share/ca-certificates/` and runs `update-ca-certificates` (picked up by Chrome and Chromium). It also imports the CA into every Firefox NSS profile found under `/home`.

> `certs/wajntraub-demo-ca.crt` is generated by `setup.sh` on the Device. If the Device and Configurator are different machines, copy this file to the Configurator first.

**Other operating systems** — import `certs/wajntraub-demo-ca.crt` manually:

| OS | Steps |
|---|---|
| Windows (Chrome) | `certmgr.msc` → Trusted Root Certification Authorities → Import |
| macOS | Keychain Access → System → File → Import Items → set Trust to "Always Trust" |
| iOS | AirDrop or email the `.crt` file → Settings → General → VPN & Device Management → Install Profile → trust it |
| Android | Settings → Security → Install from storage → CA certificate |

Restart your browser after importing.

**Firefox (any OS)** — Firefox maintains its own certificate store, separate from the OS. Even if you imported the CA into the OS store, Firefox requires a separate import:

1. Copy `certs/wajntraub-demo-ca.crt` from the Device to the machine running Firefox.
2. In Firefox: **Settings** → **Privacy & Security** → scroll to **Certificates** → **View Certificates**
3. **Authorities** tab → **Import**
4. Select `wajntraub-demo-ca.crt`
5. Check **Trust this CA to identify websites** → OK
6. Restart Firefox

> If you run `setup.sh` again, a new CA is generated. You must re-download and re-import — the name "Wajntraub Demo CA" will be the same but the key will differ, causing `SEC_ERROR_UNKNOWN_ISSUER`. Delete the old entry in the Authorities tab before importing the new one.

## Step 3 — Run

> **Run on: Device**

**Wi-Fi interface** — the provisioning daemon auto-detects the first wireless interface via `iw dev`. If your machine has multiple Wi-Fi interfaces or the wrong one is picked, find the correct name:

```bash
iw dev
```

and override it via `PROVISION_IFACE` as shown below.

**Run the provisioning daemon:**

```bash
sudo PROVISION_IFACE=<your-interface> .venv/bin/python3 device/provision.py
```

> `sudo` strips environment variables, so `PROVISION_IFACE` must be passed inline as shown above, not exported beforehand. `.venv/bin/python3` is an absolute-style path relative to the repo root — no activation needed.

Expected terminal output (happy path):

```
2026-03-31 12:00:00 INFO State: INIT
2026-03-31 12:00:01 INFO State: INIT → AP_MODE
2026-03-31 12:00:01 INFO State: AP_MODE → PROVISIONED
2026-03-31 12:00:01 INFO State: PROVISIONED → CONNECTING
2026-03-31 12:00:05 INFO State: CONNECTING → ONLINE
2026-03-31 12:00:05 INFO Connected to network: MyHomeWifi
2026-03-31 12:00:05 INFO Provisioning complete — device is connected.
```

**Provisioning steps:**

1. The Device broadcasts a Wi-Fi AP named `Wajntraub-Demo-0000` on the interface you specified
2. On the **Configurator**, connect to the `Wajntraub-Demo-0000` Wi-Fi network
3. On the **Configurator**, open a browser and navigate to `https://setup.wajntraub-demo.local` — a padlock should appear with no TLS warning
4. Enter the target Wi-Fi SSID and password, click **Provision**
5. The Device disconnects from the SoftAP, connects to the target network, and logs `ONLINE`

**Environment variables** (set on the Device before running):

| Variable | Default | Description |
|---|---|---|
| `PROVISION_IFACE` | auto-detected | Wi-Fi interface to use for SoftAP. Auto-detected via `iw dev` if unset. Override if the wrong interface is picked. |
| `PROVISION_TIMEOUT` | `600` | Seconds to wait for credential submission before giving up |

```

  provision.py        State machine (INIT → AP_MODE → … → ONLINE)
  server.py           Flask HTTPS credential server
  wifi.py             SoftAP lifecycle (hostapd + dnsmasq) and nmcli connect

  setup.sh            One-time PKI initialisation (run on Device)

  unit/               Flask + state machine unit tests (no hardware required)
docs/
  architecture.md     System design, C4 diagrams, test strategy
```

## Production deployment

This demo uses a manually invoked process and a repo-local `.venv`. For a production device fleet:

| Step | Approach |
|---|---|
| **Service lifecycle** | Replace manual invocation with a `systemd` unit. `ExecStart=/opt/wajntraub-provision/.venv/bin/python3 device/provision.py` gives auto-start on boot, restart on failure, and `journald` logging — no terminal session required. |
| **Packaging** | Build a `.deb` (using `fpm` or a `debian/` directory). The `postinst` script runs `setup.sh`, creates the venv, installs deps, and enables the systemd unit. Installation becomes `dpkg -i wajntraub-provision_1.0_arm64.deb`. |

## Security notes and limitations

**Software key for TLS:** This demo uses a software RSA key (`certs/device.key`) for TLS. In production, the private key would be generated inside a hardware TPM (e.g. Infineon SLB9670) and never exported. Python's `ssl.SSLContext` cannot load TPM keys directly — the production path is `tpm2-pkcs11`, which exposes the TPM key via PKCS#11 and is reachable from Python ssl with a `pkcs11:` URI. This is documented in `docs/architecture.md`.

**certs/ directory:** All files under `certs/` are generated locally by `setup.sh` and are excluded from version control (`.gitignore`). Never commit private keys or CA credentials.
